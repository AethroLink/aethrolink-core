# AethroLink v0.1 Status

## Bottom line

The current service satisfies the core local orchestration slice:
- submit task
- route runtime
- auto-launch if down
- stream progress
- persist history and results
- return final result

It does not yet satisfy the stronger interpretation of true agent-to-agent transport, identity, and trust as fully implemented working subsystems. Those boundaries mostly exist in types and docs, not as fully wired runtime behavior.

## What v0.1 says it should do

From the README, v0.1 is supposed to support:
- one runtime sends a task to another runtime
- progress is streamed back
- a final response is returned
- the target runtime is started automatically if it is down

From `docs/implementation_plan.md`, the acceptance gates are:
1. `go test ./...` passes
2. the HTTP server starts successfully
3. tasks can be submitted to:
   - Hermes adapter
   - OpenClaw adapter
   - HTTP ACP adapter
4. SQLite migrations run cleanly
5. event streaming works over SSE
6. history survives restart
7. fake runtimes are included in the repo
8. stub adapters compile cleanly
9. docs match the implemented contracts

## What the current service can do

### 1. Local control plane works
The HTTP API exists for:
- `POST /v1/tasks`
- `GET /v1/tasks/{id}`
- `GET /v1/tasks/{id}/events`
- `POST /v1/tasks/{id}/resume`
- `POST /v1/tasks/{id}/cancel`
- `GET /v1/runtimes`
- `GET /v1/runtimes/{id}/health`
- `POST /v1/runtimes/{id}/start`
- `POST /v1/runtimes/{id}/stop`
- `GET /artifacts/{artifact_id}`

### 2. Task routing works
A request can:
- specify `target_runtime` directly
- or resolve by `intent` and runtime capability

### 3. Auto-launch-if-down works
The orchestrator:
- checks adapter health
- emits launch-related events if the runtime is down
- launches the runtime if `launch_if_down=true`
- dispatches once ready

### 4. Runtime supervision works
The service supports:
- managed runtime process launch
- runtime lease tracking
- launch history persistence
- runtime health checks

### 5. Persistence works
The service persists:
- runtimes
- tasks
- task events
- runtime leases
- launch history
- payload/result artifact metadata in SQLite
- artifact files on disk

### 6. SSE event streaming works
Task event history can be replayed and live events stream until terminal state.

### 7. Restart history survives
After restart, persisted task history and artifact metadata remain available.

### 8. Fake adapter paths work for contract validation
The repo contains working fake paths for:
- Hermes-style ACP client runtime
- OpenClaw-style ACP client runtime
- HTTP ACP runtime

These validate orchestration and state-machine behavior.

### 9. One real Hermes slice now works
A real Hermes-backed runtime path exists via the `core` target.
It has been verified to:
- start a real Hermes ACP process
- create a real ACP session
- send a prompt
- stream output
- persist the final result

Verified real-executor examples:
- `aethrolink-agent` executor: `Say exactly OK` -> `OK`
- `mimoportal` executor: `Say exactly OK` -> `OK`

### 10. Resume and cancel APIs exist
The service supports:
- resume from `awaiting_input`
- cancel through the adapter boundary
- event logging for resume/cancel requests

## What the current service cannot do, or cannot honestly claim yet

### 1. It is not yet true runtime-to-runtime transport
What exists today:
- client -> AethroLink HTTP API -> runtime adapter

What does not yet exist as a real subsystem:
- runtime/node -> transport adapter -> another node/runtime

So if v0.1 is interpreted literally as true node-to-node or runtime-originated delegated transport, the answer is no.

### 2. Identity is boundary-only, not a real implemented feature
The type boundaries exist for identity, but the task flow does not yet include:
- active node identity lifecycle
- signing
- verification
- trust enforcement

### 3. Auth and authorization are not present
The current control plane has no:
- API auth
- scoped permissions
- approval gates
- actor attribution model for sensitive actions

### 4. Fake adapters are still mostly fixtures
The fake Hermes/OpenClaw/HTTP runtimes prove plumbing and contracts.
They do not prove production-grade behavior.

### 5. Real Hermes is still a thin slice
The real Hermes path proves:
- launch
- session creation
- prompt dispatch
- output capture
- result persistence

It does not yet prove:
- robust multi-turn continuity
- crash-safe reattachment
- security and approval boundaries
- broader production workflow support

### 6. Restart continuity is now explicit, but still local-first
What survives restart today:
- task history
- task state
- artifacts
- thread history
- thread continuation after restart for completed turns
- interrupted marking for thread-bound tasks that were left non-terminal at restart

What is still not yet guaranteed:
- exact reattachment to in-flight external runtime work
- reliable post-restart reuse of remote sessions for runtimes that only persist session state in-process

### 7. Real ACP adapter internals are now layered, but still local-first
The active adapter path is no longer just one ACP file. It is moving toward:
- local runtime host
- ACP session transport
- runtime dialects such as Hermes/OpenClaw
- runtime-neutral adapter event mapping

That improves clarity for local runtimes and keeps future transport seams open,
but it is still not yet the same thing as real node-to-node transport.

### 8. The “docs match implementation” claim is only partial
Docs are directionally aligned on orchestration, state, storage, and adapters.
But some wording is still stronger than the actual implementation, especially around:
- local loopback transport
- local identity abstraction

The interfaces exist.
The fully working subsystems do not.

## Judgment against the explicit acceptance gates

### Pass
1. `go test ./...` passes
2. the HTTP server starts
3. tasks can be submitted to Hermes, OpenClaw, and HTTP ACP adapter paths
4. SQLite migrations/storage run cleanly
5. SSE event streaming works
6. history survives restart
7. fake runtimes are included
8. adapters compile cleanly

### Partial / not fully honest to claim complete
9. docs match implemented contracts
- mostly true for orchestration/state/storage/adapters
- not fully true if transport and identity language is read literally

## Judgment against the README’s top-line v0.1 promise

### Pass if interpreted as local orchestration
The current service can:
- submit a task to a runtime
- stream progress
- return a final response
- auto-launch the runtime if it is down

### Not yet pass if interpreted as true agent-to-agent or networked delegation
The active Go implementation does not yet provide:
- a real transport layer
- a real identity and trust layer
- true inter-node runtime-to-runtime delivery

## Clean summary

The current service is:
- a working local orchestration node
- with persistence
- supervision
- routing
- SSE streaming
- fake adapter validation
- one thin real Hermes adapter

The current service is not yet:
- a real runtime-to-runtime transport network
- an authenticated control plane
- a restart-safe distributed long-running agent system

## Honest v0.1 wording

A more accurate v0.1 statement is:

> AethroLink v0.1 is a local runtime orchestration node that can route tasks to runtimes, launch them on demand, stream progress, and persist results.

## Likely v0.1.x hardening items

- auth and authorization on the control plane
- real Hermes as a first-class supported adapter
- sticky Hermes session continuity
- restart reconciliation
- real transport and identity implementation if the product wants to claim true agent-to-agent behavior beyond local orchestration
